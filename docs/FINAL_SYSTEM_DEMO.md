# RTMPose + ST-GCN++ 系统演示与滑窗冒烟验证

更新日期：2026-07-24

## 系统默认配置

- 姿态前端：RTMPose balanced。
- 分类器：四折 ST-GCN++ 模型概率平均。
- 正式权重：`results/benchmark_e300_full/rtmpose_stgcnpp/fold_*/best.pt`。
- 滑动窗口：64 个连续视频帧，步长 16 帧。
- 报警阈值：平均跌倒概率不低于 0.5，至少 3/4 个折模型同意，
  连续 3 个窗口为阳性才确认。
- 姿态有效率低于 50% 时输出 `UNKNOWN`。

演示画面显示骨架、跌倒概率、状态、路线、姿态有效率和折模型同意数。

## 正式演示视频

跌倒片段 `Subject 1/Fall/01.mp4`：

- 203 帧、9 个滑动窗口；
- 姿态缺失 0 帧；
- 产生 1 次确认报警；
- 确认时概率约 0.979，4/4 折同意，姿态有效率 1.00；
- 单路 GPU 处理速度约 20.87 FPS。

ADL 片段 `Subject 1/ADL/01.mp4`：

- 245 帧、12 个滑动窗口；
- 姿态缺失 0 帧；
- 12 个窗口全部保持 `NORMAL`，无确认报警；
- 单路 GPU 处理速度约 21.96 FPS。

输出目录为 `outputs/final_rtmpose_system/`。

## 8 个片段冒烟验证

为检查系统状态机而额外选取 4 个跌倒和 4 个 ADL 片段。除上述两段生成
完整视频外，其余片段使用 `--no-video` 运行。结果：

|类别|片段数|正确|错误|
|---|---:|---:|---:|
|Fall|4|3 次确认|1 次漏检|
|ADL|4|3 次无报警|1 次误报|

- 漏检：`Subject 4/Fall/08.mp4`，最高平均概率约 0.537，但最多只有
  2/4 折同意，未达到 3/4 的报警条件。
- 误报：`Subject 2/ADL/01.mp4`，最高平均概率约 0.788，达到 3/4 折同意
  并连续确认。
- 8 个片段均无姿态缺失。
- 多个进程并行抢占同一 GPU 时只有约 4.7–6.3 FPS；该数字不能代表单路性能。

这只是系统流水线冒烟检查，不是新的无偏准确率估计：四折模型集成中有部分模型
见过对应主体。正式模型性能仍以主体隔离四折合并指标为准。

## 暴露出的下一项算法问题

训练阶段使用“从完整片段均匀采样 64 帧”，系统阶段使用“连续 64 帧滑动窗口”，
两者存在输入分布差异。当前的 1 次漏检和 1 次误报不能只靠调整阈值解决。
下一项优先工作应是从原视频生成带时间标签的连续滑窗训练集，并按主体隔离重新训练，
使训练输入与部署输入一致。
